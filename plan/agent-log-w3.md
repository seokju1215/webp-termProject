# Agent Log — Week 3 (6월 4일 ~ 6월 10일)

## 디자인 시스템 수립

**목적**
프론트엔드 코딩 전에 색상, 폰트, 아이콘, 레이아웃, 브랜드명 등 시각적 방향성을 확정하기 위해 Agent를 활용함. 막연하게 "어떻게 할까요?" 방식으로 물어보는 것보다 인터뷰 방식으로 하나씩 결정하는 구조가 필요했음.

**요청 및 프롬프트**

| # | 요청 내용 요약 | 실제 사용한 프롬프트 (핵심 부분) | 효과적이었던 이유 |
|---|--------------|-------------------------------|----------------|
| 1 | 디자인 시스템 인터뷰 시작 | `/oh-my-claudecode:plan 디자인 시스템 정하자` | plan 스킬의 인터뷰 모드를 활용해 tone → 아이콘 → 컬러 → 폰트 → 레이아웃 → 브랜드명 순으로 하나씩 결정할 수 있었음. 한 번에 다 묻지 않아서 각 결정에 집중할 수 있었음 |
| 2 | 결정 확인 후 진행 | `응` | 확정된 결과(teal, Pretendard, Lucide, 사이드바+스플릿패널, QAFlow)를 한 번에 요약해서 보여준 뒤 동의를 받는 구조가 명확했음 |

**생성 결과 vs 실제 반영**

| 항목 | Agent 결과 | 실제 반영 | 수정 이유 |
|------|-----------|----------|----------|
| primary 컬러 | #0ca678 (teal) | 그대로 | 깔끔하고 QA 툴에 어울림 |
| 폰트 | Pretendard Variable (CDN) | 그대로 | 한국어 가독성 최우선 |
| 아이콘 | Lucide CDN (`unpkg`) | 그대로 | tree-shaking 불필요한 환경에서 CDN이 더 편함 |
| 레이아웃 | 사이드바(240px) + 콘텐츠 영역 | 그대로 | 팀 목록 상시 표시에 적합 |
| 프로젝트 상세 레이아웃 | TC 패널(380px 고정) + 칸반 패널(flex:1) | 그대로 | 테스트케이스 목록과 칸반을 동시에 보는 것이 UX상 핵심 |
| 브랜드명 | QAFlow | 그대로 | 직관적이고 기억하기 쉬움 |

**오류 사례**

- **상황**: 디자인 시스템 이야기를 꺼내자마자 Agent가 인터뷰 없이 바로 `AskUserQuestion` 한 번으로 끝내고 CSS 코드를 짜기 시작함
- **문제점**: 결정해야 할 항목이 6~7개인데 한 번의 질문으로 압축해서 물어보니 각 항목에 대한 충분한 고민 없이 넘어갈 뻔했음. 사용자가 명시적으로 거절("뒤질래")
- **수정 내용**: `/oh-my-claudecode:plan` 스킬을 명시적으로 호출해 인터뷰 모드로 재진행. 항목별로 하나씩 결정하는 구조로 바꿔서 각 선택에 이유를 붙일 수 있었음

**소감**
plan 스킬의 인터뷰 모드가 디자인 시스템 확정에 생각보다 잘 맞았다. "어떻게 할까요?" 대신 "이 중 뭐로 할까요?" 형식으로 진행되니 선택 피로가 적었고, 결과물도 이유가 명확한 결정들로 구성됐다.

## 프론트엔드 전체 구현

**목적**
디자인 시스템 확정 후 5개 HTML 페이지, 5개 JS 페이지, CSS 클래스, 공통 유틸리티를 한 번에 구현하기 위해 Agent를 활용함. 파일 수가 많아 직접 하나씩 짜기엔 시간이 많이 걸리고 일관성 유지가 어려웠음.

**요청 및 프롬프트**

| # | 요청 내용 요약 | 실제 사용한 프롬프트 (핵심 부분) | 효과적이었던 이유 |
|---|--------------|-------------------------------|----------------|
| 1 | 전체 구현 지시 | `다 해` | 이미 디자인 시스템과 API 스펙이 확정된 상태라 두 글자로도 충분했음. 컨텍스트에 충분한 정보가 있었음 |
| 2 | (직접 수정 없이) 서버 기동 검증 요청 | 암묵적 — 구현 완료 후 자동으로 서버 테스트 진행 | Agent가 완료 선언 전에 `curl`로 직접 API 응답 확인까지 해줘서 "됐다고 주장만 하는" 문제가 없었음 |

**생성 결과 vs 실제 반영**

| 항목 | Agent 결과 | 실제 반영 | 수정 이유 |
|------|-----------|----------|----------|
| `dashboard.js` | 팀 카드 렌더링 + 팀 생성 모달 | 그대로 | 모달, 스피너, 토스트 패턴이 일관성 있게 구성됨 |
| `projects.js` | 프로젝트 카드 + 이미지 업로드 + 북마크 | 그대로 | `FormData` + `apiUpload` 패턴 정확히 활용 |
| `project-detail.js` | TC 목록 + 칸반 드롭다운 + 담당자 지정 모달 | 그대로 | 드롭다운 open/close 이벤트 위임 처리가 깔끔하게 됨 |
| `mypage.js` | 프로필 조회/수정 + 북마크 + 팀 목록 | 그대로 | 북마크를 팀별 병렬 요청으로 수집하는 방식이 효율적 |
| `qa.js` | iframe + 피드백 사이드패널 + Agentation 가이드 | 그대로 | iframe 차단 시 fallback UI(`.iframe-blocked`) 처리 포함 |
| CSS 클래스 | 초기 components.css에 누락 | 직접 추가 | `.tc-item`, `.feedback-item`, `.bookmark-item`, `.stat-mini`, `.iframe-blocked`, `.qa-tab`, `.code-block` 등 JS 렌더 함수에서 참조하는 클래스가 빠져 있었음 — 검증 단계에서 발견해 일괄 추가 |
| `PUT /api/auth/me` | 프론트에서 호출하나 백엔드 라우트 없음 | routes/auth.js에 직접 추가 | 마이페이지 이름 저장 기능에 필수. Agent가 프론트 구현 후 백엔드 누락을 스스로 발견하고 추가함 |

**오류 사례**

- **상황**: 컨텍스트 압축(compaction) 전 세션에서 `project-detail.js`와 `projects.js`를 Read 없이 Edit 시도
- **문제점**: `File has not been read yet` 오류로 Edit 실패
- **수정 내용**: Read → Edit 순서로 재진행. 이후 세션에서는 Write로 전체 교체 방식 사용해 Read 필수 제약 회피

**소감**
"다 해" 두 글자로 파일 12개(HTML 5 + JS 7 + CSS 수정 + 백엔드 1)가 일관된 패턴으로 나왔다는 게 인상적이었다. 다만 CSS 클래스 누락은 JS 렌더 코드와 CSS를 별도로 생성하는 구조상 생기는 한계라, 검증 단계(`grep`으로 클래스 존재 확인)가 필수임을 확인했다.
